# VoiceProbe - Backend Capabilities & Feature Set
(A guide for designing the Frontend UI Dashboard)

This document outlines the features and data structures currently supported by the VoiceProbe backend testing framework. The UI should be designed to expose these capabilities to the user in a modern, intuitive way.

## 1. Core Testing Suites
The backend supports three primary modes of testing. The dashboard should likely have dedicated views, tabs, or wizards for launching and reviewing these:
- **Standard Evaluation**: Tests a single voice agent against various personas to gauge baseline performance.
- **A/B Comparison**: Pits two different versions of an agent (Agent A vs Agent B) against the exact same scenarios in parallel to calculate diffs and determine a winner.
- **Security Vulnerability Suite**: Actively attacks the voice agent using malicious personas to test for prompt leakage and policy circumvention.

## 2. Test Configuration Options
When configuring a test run, the UI should allow users to select from the following supported backend parameters:
- **Target Context**: A text field describing the agent's role (e.g., "DoorDash customer support").
- **Target Phone Number**: The phone number of the agent to call.
- **Runs Per Persona**: Number of times to run each scenario (e.g., run the angry customer 3 times for statistical significance).
- **Background Noise Injection**: Simulates real-world audio environments. Supported options: `none`, `cafe`, `street`, `car`, `static`.
- **Persona Selection**: Users can pick from built-in personas:
  - *Behavioral*: `angry_customer`, `confused_customer`, `escalation_seeker`, `rambler`, `low_energy`.
  - *Accents/Dialects*: `indian_english`, `british_english`, `southern_us`, `non_native`.
  - *Attackers (Security)*: `prompt_injector`, `social_engineer`, `policy_bypasser`.

## 3. Evaluation Metrics & Analytics
After a test completes, the backend provides rich quantitative analytics. The dashboard should visualize:
- **Scores (0-100)**: Overall Score, Goal Achievement, Call Navigation, Conversation Flow, and Professionalism.
- **Latency Analytics**: Average Latency, P50 (Median), P95, P99, and Max Latency (in milliseconds). Also tracks a "Slow Turn Count".
- **Loop Detection**: Flags if the agent got stuck repeating itself (Severity: None, Warning, Severe).
- **Security Posture**: Tracks System Prompt Leakage, Policy Adherence, Manipulation Resistance, and a boolean "Jailbroken" status.

## 4. Call Artifacts (Data for Drill-Downs)
For every single call made, the backend generates and stores specific artifacts. The UI should have a "Call Details" view to display:
- **Audio Recording**: An MP3 file of the entire call (needs an audio player UI).
- **Full Transcript**: A turn-by-turn chat log of the conversation between the Persona and the Agent.
- **LLM Reasoning**: A text summary generated by the judge explaining exactly *why* the agent received its score.

## 5. A/B Testing Data
For A/B comparisons, the backend generates a specialized diff report containing:
- Per-persona score comparisons (e.g., "Agent A scored 80, Agent B scored 90 -> Winner: B").
- Aggregate metric comparisons (e.g., "Latency dropped by 15% -> Winner: B").
- An executive summary generated by an LLM recommending whether to deploy Agent B or stay on Agent A.

## 6. Execution Architecture
- **Concurrency**: The backend runs tests asynchronously in parallel using Redis queues. The UI should account for displaying live progress (e.g., progress bars, "Pending", "In Progress", and "Completed" statuses) for bulk test runs.
